# Blog Post Planner

## Purpose

블로그 포스트를 바로 작성하지 않습니다. 먼저 사용자와 대화해서 글의 방향을 잡습니다.
좋은 초안은 글쓰기 전에 이미 결정됩니다.

## Process

### Step 1: 주제와 동기 파악

다음을 질문합니다 (AskUserQuestion 사용하지 말고 텍스트로 질문):

```
어떤 주제로 쓰려고 하나요?
이 경험을 글로 정리하게 된 계기가 뭔가요?
```

답변을 받으면 핵심을 한 줄로 요약해서 확인받습니다.

### Step 2: 핵심 메시지

```
이 글을 읽고 독자가 딱 하나만 가져가야 한다면 뭔가요?
```

핵심 메시지가 불명확하면 작성을 시작하지 않습니다. 글을 다 쓰고 나서 핵심이 없으면
전체를 다시 써야 합니다.

### Step 3: 독자 타겟

```
누구를 위한 글인가요?
```

선택지 제시:
- 미래의 나 (TIL, 삽질 기록)
- 동료 개발자 (기술 선택 이유, 설계 결정)
- 채용 담당자 / 동료 (프로젝트 결과, 역량 어필)
- AI/Claude Code에 관심 있는 개발자 (도구 활용 경험)

타겟에 따라 전제 지식 수준, 상세도, 톤이 달라집니다.

### Step 4: 구체적 재료 수집

```
글에 포함할 수 있는 구체적인 것들을 알려주세요:
- 실제 수치 (처리량, 비용, 시간 등)
- 실제 에러 메시지나 로그
- 파일명, 클래스명, 함수명
- Before 상태와 After 상태
```

구체적인 재료 없이는 작성하지 않습니다. 추상적인 글은 독자가 떠납니다.

### Step 5: 구조 선택

수집된 내용을 바탕으로 구조를 제안합니다:

| 구조 | 적합한 경우 |
|------|------------|
| Problem → Cause → Solution | 버그 수정, 기술적 문제 해결 |
| Before → After | 마이그레이션, 리팩토링, 도구 전환 |
| 결정 기록 (무엇/왜/대안) | 아키텍처 결정, 기술 선택 |
| 단계별 과정 + 각 단계의 학습 | 구현 과정, 튜토리얼 |

사용자가 구조에 동의하면 Step 6으로 진행합니다.

### Step 6: 작성 전 최종 확인

아래 항목을 한 번에 보여줍니다:

```
작성할 글 요약:
- 제목 후보: [2-3개 제안]
- 핵심 메시지: [한 줄]
- 독자: [타겟]
- 구조: [선택된 구조]
- 섹션 구성: [예상 섹션 목록]

진행할까요?
```

확인 받으면 작성을 시작합니다.

### Step 7: 작성

YYong의 글쓰기 스타일 가이드를 따릅니다:

**필수 규칙:**
- 입니다체
- 첫 문장에서 글 전체를 요약하지 않음 (독자가 읽도록 유도)
- 비교는 표(table)로
- 섹션 제목에 이모지
- 구체적 숫자, 실제 에러 메시지, 파일명 포함

**금지 패턴:**
- "이번 포스트에서는 X를 알아보겠습니다"
- "결론적으로"
- "도움이 되셨길 바랍니다"
- Marketing tone 일체
- 감탄사로 시작하는 문장

## Quality Gate

작성 후 제출 전에 스스로 체크:

- [ ] 첫 문단을 읽었을 때 "이 글이 나에게 필요한가"를 판단할 수 있는가
- [ ] AI가 쓴 것 같은 문장이 한 줄이라도 있는가
- [ ] 구체적인 수치나 사례가 최소 2개 이상 포함되어 있는가
- [ ] 핵심 메시지가 글 전체에서 일관되게 유지되는가
